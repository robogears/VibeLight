# moonlight-qt 6.1.0 CLI on this Mac — complete findings

Binary: `/Applications/Moonlight.app/Contents/MacOS/Moonlight` — reports `Moonlight 6.1.0` via `--version`. Every invocation prints `Redirecting log output to /tmp/Moonlight-<epochSecs>.log` to **stderr** first (Qt log messages go to that file; only 10 most recent log files are kept, 10 MB cap). CLI status/error text is printed with plain fputs to stdout/stderr, NOT to the log file. For failed `list`/`quit` runs the log file stayed 0 bytes.

## Actions (top-level `--help`)
`Usage: Moonlight [options] <action>` — actions: `list`, `quit`, `stream`, `pair`. No arguments = normal GUI launch. GOTCHA: bare `Moonlight --help-all` (no action) does NOT print help — it launches the full GUI and never returns (had to be killed). Per-action `--help`/`--help-all` works fine.

## `stream` — full flag list (from `stream --help-all`, verified live)
`Moonlight [options] stream <host> "<app>"`
- Resolution: `--720` (1280x720), `--1080` (1920x1080), `--1440` (2560x1440), `--4K` (3840x2160), `--resolution <WxH>` (regex `^(\d+)x(\d+)$`, case-insensitive; bad format = hard error exit 1)
- `--fps <n>` — int; values outside 10–480 print a warning to stderr but proceed
- `--bitrate <Kbps>` — outside 500–500000 prints warning. If you pass resolution/fps but NOT `--bitrate`, the CLI recomputes the default bitrate via `getDefaultBitrate(width,height,fps,yuv444)` instead of using the plist bitrate
- `--packet-size <bytes>` (>=1024)
- `--display-mode <borderless|fullscreen|windowed>` → windowMode enum (fullscreen=0, borderless/"fullscreen desktop"=1, windowed=2)
- `--audio-config <stereo|5.1-surround|7.1-surround>` → audioConfig 0/1/2
- `--video-codec <auto|H.264|HEVC|AV1>` → videoCodecConfig (auto=0, H264=1, HEVC=2, deprecated HEVC-HDR=3, AV1=4)
- `--video-decoder <auto|hardware|software>` → videoDecoderSelection 0/1/2
- `--capture-system-keys <never|fullscreen|always>` → captureSysKeysMode 0/1/2
- Boolean toggle pairs (`--X` / `--no-X`): vsync, multi-controller, quit-after (quit remote app when session ends), absolute-mouse (remote-desktop mouse), mouse-buttons-swap, touchscreen-trackpad, game-optimization, audio-on-host, frame-pacing, mute-on-focus-loss, background-gamepad, reverse-scroll-direction, swap-gamepad-buttons (Nintendo style), keep-awake, performance-overlay, hdr, yuv444

## `list`
`Moonlight [options] list <host>`; flags `--csv` and `--verbose`. Runs **headless** (hasGUI=false in main.cpp) — the only fully GUI-free action, and the only one with reliable exit codes. Plain output: one app name per line. `--verbose` adds "Establishing connection to PC..." / "Loading app list..." lines to stdout. `--csv` header: `Name, ID, HDR Support, App Collection Game, Hidden, Direct Launch, Boxart URL`; rows: `"%s",%d,%s,%s,%s,%s,"%s"` (name and boxart URL quoted, booleans lowercase true/false). Timeouts: computer seek 30 s, app-list seek 10 s. Exit: 0 success, -1 (=255 in shell) on failure with `Failed to connect to <host>` or `Computer <x> has not been paired...` on stderr. Verified live: unreachable paired host → stderr `Failed to connect to <HOST_IP>`, exit 255, empty stdout.

## `quit`
`Moonlight [options] quit <host>` — no other flags. Sends the NvHTTP cancel/quit-app request via ComputerManager::quitRunningApp; on Sunshine-family hosts (incl. Vibepollo) this **fully terminates the running remote app**, not just the session. Requires host already paired (`Computer %1 has not been paired.` otherwise). Computer seek timeout 10 s. On success: `QCoreApplication::exit(0)` automatically. **CRITICAL GOTCHA (verified live):** `quit` shows a small GUI window (CliQuitStreamSegue.qml, busy spinner), and on ANY failure it opens a modal error dialog and blocks FOREVER waiting for the user to click it (tested: still running 25+ s after failure, nothing on stderr — the error text goes to the dialog, not the console). After user closes the dialog, Qt.quit() → exit 0. So exit code and stderr are useless failure signals for `quit`; a wrapper must enforce its own timeout + SIGKILL, or better, quit apps via the Vibepollo/Sunshine HTTPS API directly.

## `pair`
`Moonlight [options] pair <host>` with `--pin <4-digit>`. Without `--pin` it generates a random PIN; with `--pin` the wrapper controls the PIN (so it can auto-submit the same PIN to Vibepollo's web/API pin endpoint). 10 s computer-seek timeout. Fails with `%1 is already paired` if paired. GUI-based (CliPair.qml shows the PIN).

## `stream` runtime behavior (source: app/main.cpp, app/cli/startstream.cpp, app/gui/CliStartStreamSegue.qml, StreamSegue.qml @ v6.1.0)
- The `stream` process IS the streaming client: it loads CliStartStreamSegue.qml, seeks the computer (30 s timeout), seeks the app (10 s), creates the Session, and pushes StreamSegue with **`"quitAfter": true`**. The process stays alive for the entire stream; when the session ends (user hotkey Ctrl+Alt+Shift+Q, remote app closed, connection lost), quitAfter triggers `Qt.quit()` → **exit code 0**. So "blocks until stream ends" = yes at the process level: wait() on the child = stream lifetime.
- **Exit codes are NOT meaningful for stream:** clean end = 0; error end = a modal GUI error dialog appears (e.g. "Failed to connect", "Failed to find application", "Computer X has not been paired. Please open Moonlight to pair before streaming.") and the process hangs until the user dismisses it, then exits 0 anyway. Only hard parse errors exit 1 immediately (message + full help on stderr, e.g. `Host not provided`). QML load failure = -1.
- Monitoring: stderr gives only the log-file path line; parse it (`Redirecting log output to (/tmp/Moonlight-\d+\.log)`) and tail that file for connection progress/errors. Stream start ≈ process alive past ~5–10 s + log activity.
- Killing the Moonlight process (SIGTERM) ends the client side only; the remote app keeps running on the host (session resumable). Full remote termination requires `quit` or the host API.

## Host and app name resolution
- `<host>` may be saved host name, UUID, or IP/address. Matching (app/backend/computerseeker.cpp) is **case-insensitive** on name and UUID (`toLower()` compare) plus comparison against all known addresses; it also calls `addNewHostManually(<arg>)` and starts mDNS polling, so a fresh IP works if reachable. Host names containing spaces work as a single quoted argv (verified: `list "MyServer"` resolved the saved host).
- `<app>` is a single argv element (quote spaces). Matching (startstream.cpp): `appList[i].name.toLower() == m_AppName.toLower()` — case-insensitive **exact full-string** match, no trimming, no fuzzy match, first match wins, no ambiguity handling. Unicode must match byte-for-byte (aside from case). **CRITICAL:** this Vibepollo host prefixes app names with invisible zero-width characters — e.g. "Steam Big Picture" is actually `U+200B U+200C U+200C Steam Big Picture` (hex e2808b e2808c e2808c 53 74 65 ...) — a uniqueness-tagging scheme. A wrapper MUST pass names verbatim as obtained from `list --csv` or the host API; hand-typed names will fail with `Failed to find application`.

Unspecified flags fall back to plist values. Keys present on this machine:

A wrapper can read this cache offline instead of calling `list`.

## Single-instance / concurrency
No single-instance guard anywhere in main.cpp (no QSingleApplication/shared memory/lock file) — every invocation is an independent process; CLI `list`/`quit`/`stream` all work while the GUI is open, and multiple streams can even run concurrently (host-side limits apply — Sunshine defaults to one session). Caveat: every action (even failed `list`) instantiates ComputerManager, which polls hosts and **writes updated host state back to the shared plist** (observed: plist mtime changed after CLI runs). macOS cfprefsd serializes writes so no corruption, but GUI/CLI host-state updates are last-writer-wins.

## Wrapper recipe (VibeLight)
1. Read hosts + cached apps + client cert/key from the plist (CFPreferences/plutil). Use `key`/`certificate` PEMs as TLS client identity for Vibepollo API calls; use `hosts.N.uuid` or address as the `<host>` argument.
2. Get authoritative app names via the host API or `Moonlight list --csv <host>` (headless, exit 0/255, includes app ID + Boxart URL); never construct names by hand (zero-width prefixes).
3. Launch: `Process` → `/Applications/Moonlight.app/Contents/MacOS/Moonlight stream <host> <exactAppName> [flags]`. Treat process lifetime = stream lifetime. Parse first stderr line for the log path; tail it for diagnostics. Guard startup with your own timeout (>40 s to cover 30 s + 10 s seeks) because failures show a modal dialog and never exit on their own.
4. Quit remote app: prefer the Vibepollo/Sunshine HTTPS cancel/close API; if using `Moonlight quit <host>`, wrap in a ~15 s timeout and SIGKILL on expiry (failure = eternal modal dialog, exit code always 0/none).
5. Do not rely on stream/quit exit codes for success detection; only `list` (0 vs 255) and parse errors (1) are reliable.
